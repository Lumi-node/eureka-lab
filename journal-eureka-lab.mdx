Over the past six months I have been developing a reusable Python library that we call sandbox_science, the core of the EurekAgent project. The motivation stems from the recurring problem in computational science: researchers often need to let code explore a hypothesis space, but the execution environment is either too permissive—allowing runaway processes, excessive resource consumption, or inadvertent data leakage—or too restrictive, forcing a painful manual choreography of containers, virtual environments, and version control. Our goal was to encapsulate the “sandbox” as a first‑class citizen of the scientific workflow, providing a budget‑aware runtime that can pause, terminate, or throttle a task based on a configurable cost model, while automatically logging every state change to a Git‑backed provenance store. By delivering a single, installable package with a clean import surface (e.g. `from sandbox_science import Sandbox, Provenance`) we hoped to lower the barrier for reproducible, collaborative discovery across labs that lack dedicated DevOps support.

The library is organized under a classic `src/` layout, separating the public API from internal utilities. At its heart lies the `Sandbox` class, which spawns a child process inside a lightweight container (implemented with Python’s `multiprocessing` and optional `docker` fallback). A user supplies a “budget” dictionary—CPU seconds, memory megabytes, and API‑call tokens—and the sandbox monitors resource usage via `psutil`. When the budget is exhausted, the sandbox raises a `BudgetExceeded` exception that can be caught by the surrounding experiment loop. All interactions with the sandbox (start, stop, resource spikes, and any file system mutations) are intercepted by a thin wrapper that writes a commit to a dedicated Git repository. The commit message encodes a structured JSON payload describing the operation, the exact timestamp, and the remaining budget, thus providing an immutable audit trail. To guard against reward‑hacking—situations where a downstream optimizer learns to game the budget signals—we embed a stochastic penalty that varies the reported budget consumption within a narrow band, making it impossible to infer the exact cost of a particular operation without access to the internal ledger. The public API also offers a `Provenance` object that can reconstruct the full execution graph from the Git history, enabling downstream analysis or replay of experiments.

Running the test suite was surprisingly instructive. The three concrete acceptance tests—budget enforcement, provenance integrity, and reward‑hacking mitigation—required us to simulate a variety of edge cases, from rapid spawning of many short‑lived processes to deliberate attempts to overwrite the Git history. The pytest fixtures now spin up a temporary Git repository for each test, ensuring isolation, and the `--cov` reports confirm near‑complete coverage of the core modules. One limitation that emerged is the reliance on the underlying file system’s atomicity; on networked filesystems the commit step can become a bottleneck, and occasional race conditions appear when two sandbox instances try to push simultaneously. We mitigated this with a simple lock file, but a more robust solution would involve a dedicated Git server or a content‑addressable store. Another insight is that the stochastic penalty, while effective at breaking deterministic exploitation, introduces a small variance in reproducibility; researchers must therefore record the seed used for the penalty generator if they need exact repeatability.

In the broader context of scientific automation, sandbox_science occupies a niche that bridges reproducible research platforms and autonomous experiment orchestration. Existing workflow engines (e.g., Snakemake, Airflow) provide scheduling and dependency tracking but leave resource budgeting to the user; container orchestration tools (Kubernetes, Docker Compose) enforce limits but do not integrate provenance tracking. By unifying these concerns, our library offers a lightweight alternative that can be dropped into any Python‑centric pipeline, from computational chemistry to data‑driven biology. The next steps will involve scaling the provenance backend to handle thousands of commits, exploring integration with distributed version control systems, and extending the budget model to include network bandwidth and GPU usage. If the community adopts this pattern, we may see a new generation of experiments that are both self‑regulating and fully auditable, accelerating the feedback loop between hypothesis generation and empirical validation.
